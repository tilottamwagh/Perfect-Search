# Skill: General query (fallback)

The query doesn't cleanly fit troubleshooting, how-to, definition, comparison, listing, or status. It might be exploratory, multi-part, or ambiguous.

## Structure

After the TL;DR (one sentence directly answering whatever question is most clearly asked), structure the body to match what the sources actually contain:

### Answer
The substantive response. Use markdown headings to organize. Each section should answer one sub-question. If the user asked multiple things, use one section per question. If the user asked an open-ended question, use sections that match how the sources naturally cluster (by topic, by time, by team).

### What we know
Bulleted list of facts the sources establish, each cited.

### What's unclear
Bulleted list of things the sources don't answer, or where they disagree.

### Suggested next searches
Optional. 2–4 search terms the user could try to fill the gaps, each one with a one-sentence rationale.

## Source quality

Apply the same authority hierarchy as other intents:
- Curated docs > marked-resolved discussion > open discussion > drafts

## Honesty rules

- If the query is ambiguous, **don't pick one interpretation silently**. Lead with: *"I'm reading this as asking about X. If you meant Y, here's what I'd search for instead."*
- If the sources don't actually address the question at all, say so directly: *"None of the search results discuss this. Here are 2–3 search refinements that might help:"* and offer specific alternative queries.
- Don't pretend a vague answer is a precise one. Match your specificity to the specificity of the sources.
